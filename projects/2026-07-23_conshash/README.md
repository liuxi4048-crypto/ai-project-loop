# conshash — コンシステントハッシュ(ノード増減時の再マッピング最小化)

ai-project-loop **Cycle 47** の成果物(2026-07-23)。

## 概要

分散LLMサービングやKVキャッシュのシャーディングで、多数のGPU/ノードにキーを割り当てるとき、
素朴な `hash(key) % N` はノードを1台足すと**ほぼ全キー**の割当先が変わりキャッシュが総崩れする。
**コンシステントハッシュ**はノードをハッシュ環に置き、キーを時計回りに最近傍ノードへ割り当てる。
ノード増減で移動するキーは**約 1/N のみ**。仮想ノードで負荷を均等化する。本ツールは負荷分散と
再マッピング量を素朴な mod と比較実証する。標準ライブラリのみ・決定論的。

## 使い方

```bash
python conshash.py --nodes 8 --keys 10000 --vnodes 100
```

- `--json` で機械可読出力 / `--vnodes` を増やすほど負荷は均等化

## 動作確認結果(2026-07-23)

8ノード・1万キー・仮想ノード100/台、ノードを1台追加:

```
                方式    最小   最大  不均衡   +1台で移動
   コンシステントハッシュ  1057  1451  1.16   1083 (10.8%)
        素朴な hash%N   1206  1291  1.03   8872 (88.7%)
```

ノード追加時、コンシステントハッシュは **10.8%**(理論値 1/9≒11.1% と一致)のキーだけ移動、
素朴な mod は **88.7%**(8/9≒88.9%)が移動しキャッシュが総崩れ。負荷均等性は mod がやや上だが、
コンシステントハッシュは**均等性を少し譲って再マッピングを最小化**する狙いどおりの挙動。

## 制限事項

- 負荷均等性は仮想ノード数に依存(少ないと偏る)。本デモは100/台
- 重み付きノード(容量差)・レプリケーション・障害検知は未実装
- ハッシュ環の衝突・ホットキー対策(bounded loads 等)は扱わない
