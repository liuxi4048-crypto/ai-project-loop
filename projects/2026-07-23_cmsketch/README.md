# cmsketch — Count-Minスケッチによる少メモリ頻度推定

ai-project-loop **Cycle 46** の成果物(2026-07-23)。

## 概要

エッジ/オンデバイスで大量のイベントストリームを扱うとき、全項目の厳密な頻度を保持する
メモリがない。**Count-Minスケッチ**は d本のハッシュ×w列の小さなカウンタ表だけで頻度を近似する
確率的データ構造。各項目を d セルに加算し、推定は d セルの最小値をとる。性質:
**真の頻度を下回らない(est ≥ true)・誤差は片側で有界・メモリは項目数に依らず一定**。
本ツールは実装して厳密計数と比較し、上位ヒッタの復元と誤差・メモリ削減を確認する。
標準ライブラリのみ・決定論的(hashlibでハッシュ)。

## 着想元(11_AI Archive)

- エッジ/オンザフライ計算の横断テーマ(横断フォールバック): ドローンの機上計算
  [[2026-07-23-computing-on-the-fly-navigating-a-vision-for-the-future-of-d-942b]] や、
  オンデバイス推論(Cactus Hybrid)など、**限られたメモリで流れてくるデータをその場で処理する**
  文脈。その基盤となる古典的確率データ構造 Count-Min スケッチを実装。

## 使い方

```bash
python cmsketch.py --width 256 --depth 4
```

- `--json` で機械可読出力

## 動作確認結果(2026-07-23)

ストリーム3000件・異なり1505種(ヘビーヒッタ5 + レア1500):

```
幅256×深さ4=1024カウンタ(異なりの68%):
  hot_0 真500→推506 / hot_3 真200→推206 …  誤差は +数件のみ
  過小評価なし(est≥true) ✓   上位5復元 ✓
幅64×深さ3=192カウンタ(異なりの13%):
  レア項目の過大評価は増える(+19〜25)が、過小評価なし ✓・上位5復元 ✓
```

厳密計数を持たずに上位ヒッタを正しく復元でき、推定は決して過小にならない(誤差は片側)。
メモリを 68%→13% に削っても上位ヒッタ復元は保たれ、精度とメモリのトレードオフを実証。

## 制限事項

- 点推定(頻度)専用。レンジ・分位点・異なり数(→HyperLogLog)は別のスケッチが必要
- 誤差は片側過大。ヘビーヒッタ判定に有効だが、レア項目の絶対値には不向き
- 保守的更新(conservative update)やCount-Sketch(両側)など精度改善版は未実装
